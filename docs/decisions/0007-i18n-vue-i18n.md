# 0007. Internationalization: vue-i18n from the start, Composition mode, a lint gate

## Status
Accepted (2026-07-23)

## Context

PROJECT.md §4 requires i18n from day one (pl + en); retrofitting is far more
expensive (issue #7). The first translatable UI — the state-machine screens (#5)
— was landing with hardcoded English, so the owner pulled #7 forward into that
PR rather than translate the screens twice. Four points needed deciding.

1. **Legacy vs Composition API.** vue-i18n runs in two modes; only one gives
   reactive live switching.
2. **How "no hardcoded strings" is enforced** (issue #7 acceptance criterion).
3. **Sentences with embedded markup** (`<code>headsetcontrol</code>`, udev paths)
   cannot be a single `t()` call without concatenation.
4. **System-locale detection and a persisted override** (issue #7 task 3), when
   there is no settings screen yet.

## Decision

- **vue-i18n in Composition mode** (`legacy: false`), `fallbackLocale: 'en'`.
  Components read strings through `useI18n({ useScope: 'global' })`; changing
  `i18n.global.locale.value` re-renders every `t()`, which is what makes the
  live-switch acceptance criterion true (proven by a test that flips pl↔en on a
  mounted screen).
- **`vue/no-bare-strings-in-template` (error)** from the already-installed
  `eslint-plugin-vue` — lighter than adding `@intlify/eslint-plugin-vue-i18n`.
  It fails in the editor and in CI on any literal text in a template. Technical
  constants (udev paths, the reload command, capability identifiers, the CLI
  name) render via interpolation, so they are data, not flagged copy.
- **Markup inside a sentence uses `<I18nT>`** (component interpolation with named
  slots): the message keeps the placeholder (`{tool}`, `{path}`, `{command}`) and
  the component fills it with a `<code>` element. No string concatenation, so
  word order stays the translator's to decide.
- **Catalogs live in `src/i18n/messages.ts`**, grouped by area (`common`,
  `screens.*`). `en` is the schema: `Messages = DeepStrings<typeof en>` forces
  every other locale to provide exactly the same keys (a missing key is a
  compile error), while allowing any string value. A test also asserts key
  parity at runtime.
- **System locale is auto-detected** (`resolveLocale`, a pure BCP-47 → supported
  locale function, + `detectLocale` reading `navigator.language`).
- **Persisting a chosen locale is deferred** to the settings issue. It is #7's
  task 3, not an acceptance criterion, and persisting a value nothing can change
  yet would be speculative (CLAUDE.md). `setLocale` exists for that UI to call.

## Consequences

- Every user-facing string is translatable and switches live; a new raw string
  fails lint, so the policy cannot rot.
- `src/i18n/` sits outside the 100% coverage gate on purpose: locale-detection
  glue has environment branches (no `navigator`, and the future tauri-store
  read) that jsdom cannot exercise cleanly. The pure `resolveLocale` and the
  catalogs are still fully tested.
- Adding a locale is one file plus one line in `SUPPORTED_LOCALES`; the compiler
  then lists every key the new catalog is missing.
- Until the settings screen lands, the locale follows the system and cannot be
  overridden in-app. The `missing-binary` copy is provisional pending the
  headsetcontrol-distribution decision (bundled vs `Depends:` vs the current
  install-it-yourself screen) — but changing it is one catalog edit.
